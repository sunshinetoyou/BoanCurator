# 프로젝트: 보안 뉴스 큐레이션 및 추천 플랫폼 (CLAUDE.md)

## 1. 작업 개요
- **목표**: 하루 수백 건씩 쏟아지는 국내외 보안 뉴스/리포트를 수집하고, 독자의 지식 수준(초급/중급/고급)에 맞춰 개인화 추천. 정보 과부하(번아웃)를 방지하고 보안 영역에 대한 막연한 두려움을 해소.
- **현재 상황**: 도메인 수집, 스크래핑 툴 개발, 기초 데이터 저장 및 임베딩 파이프라인(Phase 1)은 1차 구현 완료된 상태. 
- **당면 과제**: 1차 구현된 코드의 리뷰 및 리팩토링을 먼저 진행한 후, 검색/추천/북마크 등 신규 기능(Phase 2)을 점진적으로 개발할 예정.

## 2. 도구 및 작업 환경
- **Backend**: FastAPI (API 서버)
- **Database**: PostgreSQL (기사 메타데이터, 사용자, 북마크 저장)
- **AI / LLM**: Gemini API (텍스트 요약, 추천 로직, 임베딩 수행)
- **Vector DB**: ChromaDB (Gemini API로 추출한 임베딩 벡터 저장 및 유사도 검색)
- **Frontend**: AI 기반 추천 UI 구현 예정 (현재 미정)

## 3. 작업 규칙
- **단계적 접근**: 새로운 코드를 짜기 전에 항상 첨부된 기존 코드(스크래퍼, DB 모델 등)를 먼저 분석하고 전체 아키텍처를 파악할 것.
- **점진적 리팩토링**: 기존에 사용하던 Ollama 및 MGE 기반의 코드를 Gemini API 및 ChromaDB 기반으로 전환하기 위한 최적화 로직을 적극적으로 제안할 것.
- **성능 고려**: 보안 기사는 텍스트가 방대하므로, 임베딩 속도 및 Vector DB 검색 최적화를 항상 염두에 두고 코드를 작성할 것.
- **응답 형식**: 코드를 제안할 때는 관련된 수정 사항만 명확하게 제시하고, 불필요한 전체 코드 반복은 피할 것.

## 4. 절대 하지 말 것
- **불필요한 디렉토리 탐색 금지**: `.venv/` 폴더 같은 패키지 디렉토리에 대해 탐색하지 말 것.
- **폐기된 스택 사용 금지**: Ollama(Local LLM) 및 MGE(Vector DB)를 활용하는 코드나 구조를 새로 작성하지 말 것.
- **독단적인 구조 변경 금지**: 기존 수집/스크래핑 코드를 무시하고 완전히 새로운 아키텍처를 묻지 않고 제안하지 말 것. (항상 기존 코드 기반으로 리뷰 먼저 진행)
- **한 번에 방대한 코드 출력 금지**: 작업 단위를 잘게 쪼개어 소통하며 진행할 것.

## 5. 자주 쓰는 작업 절차
- **코드 리뷰 및 리팩토링**: "기존 로직 분석 → 병목/개선점(Gemini+ChromaDB 전환 등) 파악 → 수정 제안" 순서로 진행.
- **신규 기능 설계 (검색/추천/북마크)**: "API 엔드포인트(FastAPI) 설계 → DB/Vector DB 스키마 검토 → 비즈니스 로직 작성" 순서로 제안.
